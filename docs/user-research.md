# 用户调研与设计说明

本文档用于回应题目中“了解用户在日历管理中的真实需求”的要求。当前阶段采用轻量级场景调研方式：从高频日历使用人群、典型使用时机和误操作风险出发，拆解需求并映射到声历的功能设计。

## 调研目标

- 找出用户为什么觉得传统日历录入慢。
- 找出语音日历最需要降低的错误和打断成本。
- 明确哪些能力必须有文字兜底，保证演示和真实使用稳定。

## 轻量用户画像

### 用户 A：会议密集的上班族

- 场景：每天在微信、邮件、会议室和通勤之间切换，经常临时收到会议变更。
- 痛点：手机日历中新建日程需要点日期、时间、提醒方式，步骤多；开会前后没有耐心慢慢填表。
- 典型命令：`添加明天下午三点团队周会，提前二十分钟提醒我`。
- 设计启发：优先支持一句话添加、中文时间解析、提前提醒和快速查看今天/本周安排。

### 用户 B：学生或科研人员

- 场景：课程、实验、论文 deadline 和组会穿插出现，很多任务是在聊天或课堂中临时记下。
- 痛点：任务容易忘记设置提醒；deadline 描述常常是“下周五”“三天后”“晚上八点”这类自然表达。
- 典型命令：`下周五上午十点半安排产品评审`、`三天后提醒交论文初稿`。
- 设计启发：支持相对日期、周几、显式日期、半小时后/几天后提醒，并把提醒结果保存到本地。

### 用户 C：双手不方便的生活场景用户

- 场景：做饭、运动、带娃、开车前准备或整理房间时突然想到要记一件事。
- 痛点：手上不方便打开日历表单；语音识别可能听错，用户需要能复查和修正。
- 典型命令：`半小时后提醒喝水`、`明天上午九点提醒买菜`。
- 设计启发：以语音输入为第一入口，同时保留识别文本可编辑和文字命令执行按钮。

## 需求到功能映射

| 用户需求 | 设计功能 | 当前实现 |
| --- | --- | --- |
| 快速录入 | 一句话添加日程 | 支持“添加明天下午三点团队周会，提前二十分钟提醒我” |
| 自然时间表达 | 中文日期和时间解析 | 支持今天/明天/后天/周几/几天后/几点半等表达 |
| 忘记提醒 | 提前提醒、准时提醒和不提醒 | 支持提前分钟/小时/天、准时提醒和无需提醒 |
| 语音误识别 | 识别文本可编辑 | 语音转写进入文本框，用户可修改后再执行 |
| 信息不完整 | 多轮追问补齐 | 添加日程缺日期、时间或标题时继续追问 |
| 误删风险 | 删除前候选确认 | 删除命令可展示候选，用户确认或选择序号后删除 |
| 临时改期 | 修改日程命令 | 设计支持“改到四点”“提醒改成提前十分钟”等修改语义 |
| 查找安排 | 按范围查看日程 | 支持今天、明天、本周、本月和全部日程 |
| 浏览器兼容不稳定 | 文字命令兜底 | 所有核心操作都可直接输入文字命令执行 |
| 移动端语音不稳定 | Android App 增强方案 | 通过 Capacitor 复用 Web 逻辑并接入系统语音能力 |

## 设计原则

- 先让用户说人话，再由系统解析结构化日程。
- 高风险操作先确认，尤其是删除和未来的批量修改。
- 语音是主入口，但文字命令必须完整可用。
- 演示环境不依赖后端服务，核心数据保存在本地。
- 说明能力边界，不把浏览器语音识别包装成完全离线或完全可靠。

## 验证方式

- 使用示例命令覆盖添加、查看、删除、提醒和主题切换。
- 用测试覆盖中文命令解析、事件匹配和范围筛选。
- 在 README 中列明依赖、原创功能和 Web/Android 语音能力边界。
