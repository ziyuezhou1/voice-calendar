# 真实需求调研与设计说明

声历面向的不是“会用日历的人”，而是经常在移动、开会、做家务、通勤或临时想到事情时，需要快速记录安排的人。设计重点是减少录入步骤、降低误识别风险，并让提醒在演示环境中可复现。

## 轻量用户画像

### 用户 A：会议密集的上班族

- 场景：一天内频繁收到会议、评审、沟通邀请，经常边走路边记日程。
- 痛点：手机日历创建事件需要多次点选日期、时间和提醒，临时记录成本高。
- 需要：一句话添加日程，能快速查看今天和本周安排。

### 用户 B：学生或科研人员

- 场景：课程、作业 deadline、组会和实验安排分散在聊天软件、邮件和口头通知中。
- 痛点：容易忘记设置提醒，临时 deadline 容易漏掉。
- 需要：支持“半小时后”“下周五上午十点半”“提前二十分钟提醒”等自然表达。

### 用户 C：双手不方便的生活场景用户

- 场景：做饭、运动、收拾东西、通勤时突然想到待办。
- 痛点：不方便解锁手机并完成复杂表单，语音识别失败时也需要兜底。
- 需要：语音优先，但识别文本可编辑，必要时可直接用文字命令执行。

## 需求到功能映射

| 真实需求 | 产品功能 | 设计取舍 |
| --- | --- | --- |
| 快速录入 | 一句话添加日程 | 优先支持中文自然语言，不要求用户按固定格式输入 |
| 信息不完整 | 多轮追问缺少日期、时间或标题 | 不强行猜测关键字段，减少错误日程 |
| 语音误识别 | 识别文本可编辑、文字命令兜底 | Web Speech API 不稳定时仍能完成核心任务 |
| 忘记提醒 | 支持提前提醒、准时提醒、相对时间提醒 | “1 分钟后提醒喝水”按目标时间提醒，不再额外提前 |
| 误删误改风险 | 删除/修改前展示候选并要求确认 | 语音删除和修改都属于高风险操作，必须可复核 |
| 快速复查 | 今天、本周、本月、全部日程视图 | 覆盖最常见的日历回看方式 |
| 节假日判断 | 月历标注法定节假日、调休和国际节日 | 帮用户判断会议和安排是否落在特殊日期 |
| 多端演示 | Web 版 + Android App 版 | Web 保留低门槛访问，Android 提供移动端语音增强方案 |
| 账号数据保留 | Google/微信演示登录 + 账号级本地存储 | 先验证账号隔离体验，后续可替换为真实 OAuth 和云同步 |

## 关键设计原则

1. 语音是入口，不是唯一入口。语音失败时，用户仍然可以编辑识别文本或直接输入命令。
2. 关键操作先确认。删除、修改这类不可逆或高风险操作，必须展示候选后再执行。
3. 解释系统边界。提醒和语音识别都受浏览器、系统和页面打开状态影响，需要在 README 中明确说明。
4. 保持可演示。核心解析、日程增删改查、账号隔离和文字命令都应能在本地或 GitHub Pages 上复现。

## 后续验证方向

- 记录 5 到 10 位目标用户最常说的添加、修改、删除命令，补充中文解析测试。
- 对比 Web 版和 Android 版的语音输入成功率，明确推荐演示环境。
- 观察用户是否理解候选确认流程，必要时把候选卡片改成可点选。
- 验证提醒弹窗、语音播报、系统通知在不同浏览器中的表现。
